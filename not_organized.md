### **Phase 2: The Tools (LangChain)**

We will separate **Internal Calculations** (Formulas) from **External Context** (APIs).

1. **`InternalMetricTool` (The Formula Engine)**
    - **Function:** Calculates internal KPIs using BigQuery SQL.
    - **Mechanism:** It maps a metric key (e.g., `net_revenue_per_active_user`) to a pre-validated SQL formula or Python calculation.
    - *Why:* Ensures the dashboard numbers are 100% accurate and match your finance reports.
2. **`ExternalMarketTool` (The Context Engine)**
    - **Function:** Calls external APIs (crypto prices, news sentiment, competitor rates).
    - **Mechanism:** Maps keys (e.g., `global_macro_volatility`) to API endpoints (e.g., Bloomberg/CoinGecko).
    - *Why:* Needed for "Market Alignment" (Box 3) to explain if a drop is internal or global.

### **Phase 3: The Nodes (The Workflow)**

This flow handles the logic of "If Dashboard → Batch Fetch" vs "If Chat → Deep Dive."

### **Node A: The Dispatcher (Entry Point)**

- **Logic:** Checks `request_type`.
    - If `dashboard_load`: Routes to **Node B (Batch Loader)**.
    - If `chat_query`: Routes to **Node C (Intent Classifier)**.

### **Node B: Batch Loader (Fast Path)**

- **Role:** Populates the 6 Domain Health Cards.
- **Logic:** Automatically triggers `InternalMetricTool` for the 6 headline metrics (ARPU, VIP Retention, Market Share, Margin Call Acceleration, Order Rejection Rate, Revenue Attribution).
- **No LLM Reasoning:** Pure data fetching for speed.

### **Node C: Intent Classifier (Chat Path)**

- **Role:** Maps user questions to the **18 High-Impact Insights**.
- **Logic:** Uses LLM to decide: "User asked about 'losing money' → This matches Insight #17 (Revenue Drop Root Cause)."
- **Output:** Sets the `mapped_insight_id` and defines which metrics are needed to answer it.

### **Node D: The Hybrid Data Fetcher**

- **Role:** The "Smart" Collector.
- **Logic:** Iterates through the `required_metrics` list determined in Node C.
    - *Is this metric internal?* → Run SQL Formula.
    - *Is this metric external?* → Call API.
    - *Is it a complex derivation?* → Run Python Math function (e.g., `correlation(internal_vol, external_vol)`).

### **Node E: The Analyst (Root Cause Engine)**

- **Role:** Connects the dots (Box 6 logic).
- **Logic:** Takes the internal data and external context to generate the narrative.
    - *Input:* "Internal Revenue: -5%", "Market Volatility: -15%".
    - *LLM Reasoning:* "Revenue dropped, but we outperformed the market drop of 15%. This is actually a positive signal of resilience."
- **Output:** Populates `root_cause_analysis`.

```python
class AgentState(TypedDict):
    request_type: str                  # "dashboard_load" or "chat_query"

# 2. DEFINE THE TOOLS (Formulas vs. APIs)
# ==========================================

# TOOL SET A: Internal Formulas (SQL/BigQuery)
@tool
def internal_metric_tool(metric_key: str) -> float:
    """Calculates internal company metrics using defined SQL formulas."""
    
    # In production, this dictionary would map to actual SQL files or dbt models
    formula_map = {
        "internal_arpu": "SELECT sum(revenue) / count(active_users) FROM metrics_daily",
        "internal_churn": "SELECT count(lost_users) / count(total_users) FROM metrics_monthly",
        "internal_revenue": "SELECT sum(total_revenue) FROM finance_gold",
        "internal_latency": "SELECT avg(execution_time_ms) FROM tech_logs"
    }
    
    if metric_key not in formula_map:
        return None
        
    print(f"   [Internal Tool] Running SQL: {formula_map[metric_key]}...")
    # Mocking a BigQuery execution result
    mock_db_results = {
        "internal_arpu": 105.2,
        "internal_churn": 0.04, # 4%
        "internal_revenue": 1500000,
        "internal_latency": 45 # ms
    }
    return mock_db_results.get(metric_key, 0.0)

# TOOL SET B: External Context (APIs)
@tool
def external_market_tool(metric_key: str) -> float:
    """Fetches external market data via APIs (Bloomberg, CoinGecko, etc.)."""
    
    api_map = {
        "external_btc_price": "https://api.coingecko.com/simple/price?ids=bitcoin",
        "external_volatility_vix": "https://api.bloomberg.com/vix",
        "external_competitor_rate": "https://competitor-scraper/rates"
    }
    
    if metric_key not in api_map:
        return None

    print(f"   [External Tool] Calling API: {api_map[metric_key]}...")
    # Mocking an API response
    mock_api_results = {
        "external_btc_price": 64500.00,
        "external_volatility_vix": 18.5,
        "external_competitor_rate": 0.02
    }
    return mock_api_results.get(metric_key, 0.0)

# ==========================================
# 3. NODE D: THE HYBRID FETCHER
# ==========================================
def node_hybrid_fetcher(state: AgentState) -> AgentState:
    """
    Iterates through required metrics and routes them to the correct tool.
    """
    print(f"\n--- NODE D: Fetching Data for {len(state['required_metrics'])} metrics ---")
    
    current_data = state.get("fetched_data", {})
    errors = state.get("errors", [])
    
    for metric in state["required_metrics"]:
        try:
            val = None
            
            # ROUTING LOGIC: Check prefix to decide tool
            if metric.startswith("internal_"):
                val = internal_metric_tool.invoke(metric)
                
            elif metric.startswith("external_"):
                val = external_market_tool.invoke(metric)
            
            # DATA MERGING
            if val is not None:
                current_data[metric] = val
            else:
                errors.append(f"Metric '{metric}' not found in any tool map.")
                
        except Exception as e:
            errors.append(f"Failed to fetch {metric}: {str(e)}")
    
    # Update State
    return {
        "fetched_data": current_data,
        "errors": errors
    }
```

### How This Works in Your Architecture

1. **The Routing Logic:** I used a simple naming convention (`internal_` vs `external_`) to decide which tool to use. This is faster and more reliable than asking an LLM to decide for every single metric.
2. **The "Formula" Layer:** The `internal_metric_tool` represents your SQL layer. Even though we are using Python here, the strings inside `formula_map` would be your actual DBT-governed SQL queries.
3. **The "Context" Layer:** The `external_market_tool` handles the APIs. This ensures that when you ask "Why did we drop?", the system has the *market context* (e.g., "BTC price dropped") ready for the analysis phase.

```python
import operator
from typing import TypedDict, List, Dict, Any, Optional, Annotated, Union
from langgraph.graph import StateGraph, END

# --- 1. STATE DEFINITION -----------------------------------------------------
# The shared memory that passes between all nodes.
class AgentState(TypedDict):
    # Inputs
    request_type: str                  # "dashboard_load" or "chat_query"
    user_query: Optional[str]          # User's question (if chat)
    
    # Classification & Routing
    mapped_insight_id: Optional[int]   # ID 1-18 (e.g., 17 for "Revenue Drop")
    insight_name: Optional[str]        # Human readable name
    
    # Data Layer
    required_metrics: List[str]        # Keys to fetch: ["internal_revenue", "external_btc"]
    fetched_data: Dict[str, Any]       # The results: {"internal_revenue": 100}
    
    # Output
    final_response: Dict[str, Any]     # The final JSON for the UI
    errors: List[str]                  # Error log

# --- 2. TOOLS (THE FORMULA & API ENGINE) -------------------------------------

def run_internal_sql_formula(metric_key: str) -> float:
    """Simulates running a dbt-governed SQL query on BigQuery."""
    # In production, this executes: client.query(SQL_MAP[metric_key])
    print(f"   [Tool: SQL] Calculating '{metric_key}'...")
    
    mock_db = {

        "internal_latency": 45,        # ms
        "internal_vip_retention": 0.94 # 94%

def call_external_market_api(metric_key: str) -> float:
    """Simulates fetching data from Bloomberg/CoinGecko."""
    print(f"   [Tool: API] Fetching '{metric_key}'...")
    
    mock_api = {
        "external_btc_price": 64500.00,
        "external_volatility_vix": 18.5,
        "external_competitor_rate": 0.02
    }
    return mock_api.get(metric_key, 0.0)

# --- 3. NODES (THE WORKERS) --------------------------------------------------

def node_dispatcher(state: AgentState):
    """Decides if this is a fast dashboard load or a complex chat query."""
    print(f"\n--- 1. DISPATCHER: Received {state['request_type']} ---")
    # No state update needed, just passing through to conditional edge
    return {}

def node_batch_loader(state: AgentState):
    """FAST PATH: Loads the 6 key metrics for the Domain Health Overview."""
    print("\n--- 2A. BATCH LOADER: Fetching Overview Cards ---")

        "internal_arpu", "internal_vip_retention", 
        "internal_revenue", "internal_latency"
    ]
    
    data = {}
    for m in overview_metrics:
        data[m] = run_internal_sql_formula(m)
        
    return {
        "fetched_data": data,
        "final_response": {"type": "dashboard_view", "cards": data}
    }

def node_intent_classifier(state: AgentState):
    """CHAT PATH: Maps natural language to one of the 18 Insights."""
    print(f"\n--- 2B. CLASSIFIER: Analyzing '{state['user_query']}' ---")
    
    # SIMULATED LLM LOGIC
    # User asked: "Why did we lose money?" -> Maps to Insight #17
    insight_id = 17
    insight_name = "Revenue Drop Root Cause"

        "insight_name": insight_name,
        "required_metrics": required

workflow = StateGraph(AgentState)

# Add Nodes
workflow.add_node("dispatcher", node_dispatcher)
workflow.add_node("batch_loader", node_batch_loader)
workflow.add_node("classifier", node_intent_classifier)
workflow.add_node("fetcher", node_hybrid_fetcher)
workflow.add_node("analyst", node_analyst)

workflow.set_entry_point("dispatcher")

def route_request(state):

        return "batch_loader"
    else:

workflow.add_conditional_edges(
    "dispatcher",
    route_request,
    {
        "batch_loader": "batch_loader",
        "classifier": "classifier"
    }
)

# Define Linear Flows
workflow.add_edge("batch_loader", END)          # Dashboard load ends here
workflow.add_edge("classifier", "fetcher")      # Chat flows to fetcher...
workflow.add_edge("fetcher", "analyst")         # ...then to analyst...
workflow.add_edge("analyst", END)               # ...then ends

# Compile
app = workflow.compile()

# --- 5. EXECUTION EXAMPLE ----------------------------------------------------

    print("\nFINAL OUTPUT JSON:")
```

### **1. What is Missing? (The Gap Analysis)**

1. **The "Math" Layer is Implicit, not Explicit:**
    - *Current State:* The code assumes the SQL tool returns the final number.
    - *The Gap:* You mentioned "mathematical formulas" (e.g., comparing volatility). If you need to calculate `(External_VIX / Internal_Volatility) * 100`, you can't do that in the SQL tool (because VIX comes from an API).
    - *Fix:* We need a **"Calculator" step** after data fetching to perform Python-math on the combined dataset.
2. **Lack of Chat Memory:**
    - *Current State:* The `AgentState` only has `user_query`.
    - *The Gap:* If the user says "Why did it drop?", gets an answer, and then asks "**What about last week?**", the current bot won't know what "it" refers to.
    - *Fix:* Add `chat_history` to the state.
3. **Structured Output Validation:**
    - *Current State:* We are simulating the LLM. In real life, LLMs sometimes return bad JSON (e.g., missing a quote).
    - *Fix:* We should use **Pydantic Models** to force the LLM to output the exact structure we need for the `Classifier`.
4. **Error Recovery:**
    - *Current State:* If the API fails, we just append an error string.
    - *Fix:* The "Analyst" node needs to see those errors and say, "I couldn't access Bloomberg, but here is what I know based on internal data," rather than crashing.

    ```python
            "final_response": {"type": "dashboard_view", "data": data}
        }
    
    def node_classifier(state: AgentState):
        """
        Identifies the Insight ID and the metrics needed.
        Includes context from chat_history if needed.
        """
        query = state["user_query"]
        print(f"--- CLASSIFIER: Processing '{query}' ---")
        
        # SIMULATED LLM DECISION
        # "Why did we lose money?" -> Insight 17
        
        return {
            "mapped_insight_id": 17,
            "insight_name": "Revenue Drop Root Cause",
            # Note we ask for raw metrics here
            "required_metrics": ["internal_revenue", "internal_volatility", "external_market_volatility"]
        }
    
    def node_hybrid_fetcher(state: AgentState):
        """Fetches Raw Data (SQL + API)."""
        print(f"--- FETCHER: Getting {state['required_metrics']} ---")
        
        fetched = {}
        errors = []
        
        for m in state["required_metrics"]:
            val = None
            if m.startswith("internal_"):
                val = run_internal_sql(m)
            elif m.startswith("external_"):
                val = call_external_api(m)
                
            if val is not None:
                fetched[m] = val
            else:
                errors.append(f"Missing data for {m}")
                
        return {"fetched_data": fetched, "errors": errors}
    
    def node_math_engine(state: AgentState):
        """
        NEW NODE: Performs Python Math on fetched data.
        This solves your requirement for 'complex formulas' combining API + SQL.
        """
        print("--- MATH ENGINE: Computing Derived Metrics ---")
        
        fetched = state["fetched_data"]
        derived = {}
        
        # Example Formula: Relative Volatility (External / Internal)
        # This explains if the market is crazier than our internal system
        if "external_market_volatility" in fetched and "internal_volatility" in fetched:
            try:
                ext_vol = fetched["external_market_volatility"]
                int_vol = fetched["internal_volatility"]
                # The Formula:
                ratio = ext_vol / int_vol if int_vol > 0 else 0
                derived["volatility_ratio"] = round(ratio, 2)
            except Exception as e:
                print(f"Math Error: {e}")
    
        return {"derived_data": derived}
    
    def node_analyst(state: AgentState):
        """
        Generates the Narrative using Raw + Derived Data.
        """
        print("--- ANALYST: Generating Insight ---")
        
        # The LLM now sees the Math results too!
        data_context = {**state["fetched_data"], **state["derived_data"]}
        
        # SIMULATED LLM RESPONSE
        narrative = {
            "headline": "Revenue Stable Relative to Market Volatility",
            "analysis": f"While revenue is flat, the Market/Internal Volatility Ratio is {data_context.get('volatility_ratio')} (High). This indicates we are shielding users from broader market chaos.",
            "sentiment": "positive"
        }
        
        return {"final_response": narrative}
    
    # ==========================================
    # 4. THE GRAPH CONSTRUCTION
    # ==========================================
    
    workflow = StateGraph(AgentState)
    
    # Add Nodes
    workflow.add_node("dispatcher", node_dispatcher)
    workflow.add_node("batch_loader", node_batch_loader)
    workflow.add_node("classifier", node_classifier)
    workflow.add_node("fetcher", node_hybrid_fetcher)
    workflow.add_node("math_engine", node_math_engine) # Added Math Step
    workflow.add_node("analyst", node_analyst)
    
    # Set Entry Point
    workflow.set_entry_point("dispatcher")
    
    # Routing Logic
    def route_request(state):
        return "batch_loader" if state["request_type"] == "dashboard_load" else "classifier"
    
    workflow.add_conditional_edges("dispatcher", route_request, ["batch_loader", "classifier"])
    
    # Chat Flow
    workflow.add_edge("classifier", "fetcher")
    workflow.add_edge("fetcher", "math_engine") # Fetch -> Math
    workflow.add_edge("math_engine", "analyst") # Math -> Analyze
    workflow.add_edge("analyst", END)
    workflow.add_edge("batch_loader", END)
    
    app = workflow.compile()
    
    # ==========================================
    # 5. EXECUTION
    # ==========================================
    
    if __name__ == "__main__":
        print("\n>>> SCENARIO: User asks complex question requiring Math <<<")
        
        inputs = {
            "request_type": "chat_query",
            "user_query": "How is our volatility compared to the market?",
            "chat_history": [], # Initialize empty memory
            "errors": []
        }
        
        result = app.invoke(inputs)
        
        print("\n>>> FINAL DASHBOARD JSON <<<")
        print(json.dumps(result["final_response"], indent=2))
    ```
    
    ### **Why this version is better:**
    
    1. **The `node_math_engine`:** This is the specific place where you put your Python formulas that need to mix SQL data and API data. It prevents the LLM from having to do math (which LLMs are bad at).
    2. **Explicit Data Flow:** You can clearly see `Fetcher` (Get Raw Numbers) $\rightarrow$ `Math Engine` (Calculate Ratios) $\rightarrow$ `Analyst` (Explain it).
    3. **Memory Ready:** The `chat_history` field is there, so when you integrate the real frontend, you can pass the previous messages back into the graph.
